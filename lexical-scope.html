🗺️ Lexical Scope

Lexical Scope

Scope determined by where code is written, not where it is called

Lexical scope means a variable's accessibility is determined at author time — by where the function is written in the source code, not where or how it is called at runtime. JavaScript resolves variables by walking outward through enclosing scopes.
Example 1

Scope Chain

Inner functions can read variables from all enclosing scopes

const level1 = 'global';

function scopeA() {
  const level2 = 'scopeA';
  function scopeB() {
    const level3 = 'scopeB';
    console.log(level3); // own scope
    console.log(level2); // outer scope
    console.log(level1); // global scope
  }
  scopeB();
}
scopeA();
✓ Output
💡 What happened?

JS walks outward through the scope chain until it finds each variable. scopeB can see its own scope, scopeA's scope, and the global scope — all at once.

Example 2

Lexical vs Dynamic Scope

Scope is locked where the function is WRITTEN, not where it is called

const lang = 'JavaScript';

function printLang() {
  // Scope locked at definition time — not call time
  console.log(lang);
}
function wrapper() {
  const lang = 'Python'; // does NOT affect printLang
  printLang();           // still prints 'JavaScript'
}
wrapper();
✓ Output
💡 What happened?

printLang was defined in the global scope, so it reads global lang='JavaScript'. The local lang='Python' inside wrapper() is invisible to it — lexical scope wins.

Example 3

Block Scope — let vs var

let and const create a new scope per {} block

function example() {
  let x = 'function scope';
  if (true) {
    let x = 'block scope'; // new x — shadows outer x
    console.log('Inside block: ' + x);
  }
  console.log('Outside block: ' + x); // outer x unchanged
}
example();
✓ Output
💡 What happened?

The inner let x is a completely separate variable from the outer x. They shadow each other within their respective blocks but never interfere. var would NOT create a new scope here — it would share the same x.

Example 4

Temporal Dead Zone (TDZ)

const/let are hoisted but not initialized — accessing early crashes

const name1 = 'global';

function sample() {
  try {
    console.log(name1); // TDZ — crashes before const line
    const name1 = 'local';
  } catch(e) {
    console.log('TDZ Error: ' + e.message);
  }
}
sample();

function fixed() {
  const name1 = 'local'; // declare FIRST
  console.log(name1);    // works fine
}
fixed();
✓ Output
💡 What happened?

JS sees const name1 inside sample() and reserves that name for the whole function scope. But it's uninitialized until that line. It won't fall back to the outer name1 — it throws a ReferenceError instead. The fix: always declare before you use.
