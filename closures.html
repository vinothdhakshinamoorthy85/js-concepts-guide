🔒 Closures

Closures

Functions that remember their outer scope even after it has closed

A closure is a function that retains access to variables from its outer (enclosing) scope even after that outer function has finished executing. The inner function carries a "backpack" of the surrounding environment wherever it goes.
Example 1

Basic Closure — Counter

A function that remembers its outer variable after the outer function returns

function makeCounter() {
  let count = 0; // closed-over variable
  return function() {
    count++;
    return count;
  };
}
const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
✓ Output
💡 What happened?

Even after makeCounter() returns, the inner function still has access to count. Every call to counter() reads and increments the same closed-over variable — that's a closure!

Example 2

Private State — Encapsulation

Closures hide variables from the outside world

function createAccount(initial) {
  let balance = initial; // private — not accessible outside
  return {
    deposit(n) { balance += n; console.log('Balance: ' + balance); },
    getBalance() { return balance; }
  };
}
const acc = createAccount(100);
acc.deposit(50);
console.log('Final: ' + acc.getBalance());
console.log('Direct access: ' + acc.balance); // undefined!
✓ Output
💡 What happened?

balance lives inside createAccount() and is never exposed. deposit() and getBalance() access it through closure. acc.balance from outside returns undefined — truly private!

Example 3

Function Factory — Multiplier

Each call creates a new closure with its own captured state

function multiplier(factor) {
  return (n) => n * factor; // closes over factor
}
const double = multiplier(2);
const triple = multiplier(3);
console.log('double(5) = ' + double(5));
console.log('triple(5) = ' + triple(5));
console.log('Same fn? ' + (double === triple));
✓ Output
💡 What happened?

Each call to multiplier(factor) returns a new function with its own factor. double and triple are independent — each closes over a different value.

Example 4

Classic Gotcha — Loop + var vs let

var shares one binding; let creates a fresh one per iteration

// var: all callbacks share the same i (function-scoped)
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log('var: ' + i), 10);
}
// let: each iteration gets its own j (block-scoped)
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log('let: ' + j), 50);
}
✓ Output
💡 What happened?

With var, all setTimeout callbacks share the same i. By the time they run, the loop is done and i=3. With let, each iteration creates a new block-scoped j that each callback closes over independently.
